# Block-Relief Outcome Summary and Reserve Pool Mechanism

## 1. Overview

This document summarizes the updated outcome reporting and reserve pool design for the Block-Relief flood relief distribution system. The project now includes a more readable, full-width demographic outcome visualization and a smart contract-backed reserve pool for any donations that exceed the campaign target.

## 2. Outcome Report Enhancements

### 2.1 Demographic Distribution Chart

The outcome report now displays the demographic distribution chart in a dedicated, larger section rather than a cramped small card. This chart is generated by the Flask backend and rendered in the frontend as a high-resolution, full-width image so it can be viewed clearly in the thesis.

### 2.2 Included Visualizations

The outcome report now highlights the following visual components:

- **Demographic Distribution Chart**
  - Visualizes gender, economic class, and vulnerability breakdowns.
  - Shows how funds were distributed across key socio-economic groups.

- **Socio-Economic Class Breakdown**
  - Table/chart of housing classes with counts, average vulnerability scores, and allocated funds under Fuzzy AHP versus traditional distribution.
  - Illustrates fund redistribution effects for `Kutcha`, `Semi-pucca`, and `Pucca` households.

- **Poverty Index Class Breakdown**
  - Analysis of fund allocation across poverty classes such as Extremely Poor, Moderately Poor, and Relatively Better-off.
  - Compares average Fuzzy AHP allocation to a flat equal distribution.

- **Geographic (Upazila) Allocations**
  - Region-wise allocation summary showing how funds were distributed across administrative areas.

- **Profile Comparison Tool**
  - A real-life perspective comparison between a fixed 10,000 BDT distribution and the dynamic Fuzzy AHP allocation for representative household profiles.

## 3. Reserve Pool Design

### 3.1 Problem Addressed

In real-life relief campaigns, donations often exceed the planned target. Without a proper mechanism, excess funds can be mixed into the current distribution pool and distort the intended allocation strategy.

### 3.2 Proposed Solution

The Block-Relief contract now tracks excess donations separately using a `reservePool`:

- The contract still uses `targetFund` as the campaign goal.
- When total donations exceed this target, the surplus is not immediately used in the current victim and lottery distribution.
- Instead, the extra amount is held in `reservePool` for future NGO support, contingency spending, or additional relief tasks.

### 3.3 Reserve Pool Behavior

At distribution time, the contract computes:

- `reserved = max(contractBalance - targetFund, 0)`
- `allocated = contractBalance - reserved`
- `lotteryPool = allocated * 20%`
- `victimPool = allocated - lotteryPool`

This ensures the original distribution target remains stable and the donated reserve is preserved separately.

## 4. Admin Control for Reserve Withdrawal

The frontend admin panel now includes a reserve withdrawal button in the `Distribute` tab. This button does the following:

- Calls `withdrawReserve()` on the smart contract.
- Sends the reserve funds to the connected admin wallet.
- Requires admin authorization and only works when reserve funds exist.

### 4.1 Design Intent

The purpose of this button is not to send funds to an arbitrary external address on the fly. Instead, it allows the authorized admin to withdraw reserve funds into the admin account, which can later be used for legitimate NGO or relief-related expenditures.

### 4.2 Practical Use Case

This approach models a real-world relief operation where:

- core campaign funding is used immediately for flood victims and lottery incentives,
- surplus donations are preserved for future emergency support,
- the administrator can later allocate the reserve for additional overhead, logistics, or follow-up relief.

## 5. Thesis Integration Notes

The new report structure is suitable for thesis presentation because it:

- improves readability of the demographic outcome visuals,
- separates immediate distribution from reserve accounting,
- introduces a practical overflow handling mechanism,
- supports a stronger fairness narrative through measurable graphs.

## 6. Summary

The updated system now combines transparent outcome reporting with a realistic reserve pool mechanism. Excess donations are no longer immediately mixed into the main distribution pool, and the reserve can be managed by the admin for additional NGO/relief needs. This provides a better real-world model and a stronger foundation for thesis discussion.
